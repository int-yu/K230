# K230 红线巡线模块设计

面向 2021 年全国大学生电子设计竞赛 F 题「智能送药小车」。走廊地面居中一条红色
胶带引导线，路口贴黑色数字纸标识病房号，药房和病房门口是黑白相间虚线。本设计只
负责**沿红线连续行驶所需的横向偏差**和路口提示。

已有的 `road.py` 负责 T/十字/END 的符号识别，本模块不替代它，也不调用它。

本文档在实拍标定后重写过。被实测推翻的初版结论保留在末尾「设计过程中被推翻的
假设」一节，避免以后重复踩坑。

## 目标与非目标

目标：

- 从单帧 RGB 图像求出红线在 5 个不同前瞻距离上的横向偏差。
- 给出路口标志和左右分支提示。
- 通过现有 `TrackingUART` 的帧格式发给 TI 板。
- 单帧耗时明显低于 `road.py`。

非目标：

- 不做 PID、不做速度规划，控制律留在 TI 板。
- 不算航向角。它完全由 `b4 - b0` 决定，不含新信息，放在 K230 算等于把控制逻辑
  挪进视觉板。TI 板需要时自行相减。
- 不做数字识别。本次只做出路口触发和状态机骨架，数字分支留接入点。
- 不保留上一帧结果做预测或时间平滑。单帧无结果就返回 `None`。

## 红色提取

RGB 通道差分，不做 HSV 转换：

```text
红 = (R - G > LINE_RED_MIN_DIFF) 且 (R - B > LINE_RED_MIN_DIFF) 且 (R > LINE_RED_MIN_VALUE)
```

`road.py` 用绿通道 Otsu 把红和黑**合并**为前景，那是符号识别需要的。巡线需要
相反的能力：必须把红线和黑色数字纸、黑色墙线分开。通道差分天然做到这点。

uint8 减法会下溢，实现必须用 `cv2.subtract`（负数饱和到 0），不能直接用 `-`。

**阈值由 `pic/line` 的 29 张实拍图标定**：

| | R−G | R−B | R |
| --- | --- | --- | --- |
| 红线像素（55 万个） | p1=61 p50=106 | p1=61 p50=106 | p1=92 p50=138 |
| 非红像素（白布、黑线、数字、污渍） | p50=−25 p99=0 | p50=−17 p99=8 | — |

场地白布在实拍中偏冷，`R−G` 是负值，与红线的 +106 之间是很宽的空白区间。阈值
扫描结果：`t=40` 保留 100.0% 误检 0.058%，`t=50` 保留 99.8% 误检 0.010%，
`t=60` 保留 98.8% 误检 0。取 `50`。

不做形态学开闭运算。反光和褶皱造成的细小断点由 run 合并时的 `LINE_RUN_MAX_GAP`
吸收，省掉每帧的形态学开销。

## 感兴趣区域

只处理画面下部 `LINE_ROI_TOP_RATIO`~`LINE_ROI_BOTTOM_RATIO`（默认 `0.32`~`1.00`）
这一段，**整体裁剪一次、缩放一次**，得到 `160×60`，再按行切成 5 条带，每条 12 行。
带与带之间**没有空隙**，水平的路口横线不会掉进缝里。

带 0 是最下面、也就是离车最近的一条。

上沿定在 `0.32` 的依据：实拍图中路口横线出现在画面高度的 `0.35`~`0.63`。上沿
再往下就会看不见远处路口，再往上则远端透视压缩严重且收益很小。ROI 上沿扫描结果：

| roi_top | 路口 mass 范围 | 终点 mass 范围 | 阈值 1.5 召回 |
| ---: | --- | --- | --- |
| 0.24 | 1.62~5.54 | 1.07~1.35 | 25/25 |
| 0.28 | 1.47~5.87 | 1.07~1.35 | 24/25 |
| **0.32** | **1.77~6.06** | **1.05~1.35** | **25/25** |
| 0.36 | 1.41~5.03 | 1.05~1.22 | 24/25 |

### 缩放方式必须避开 INTER_AREA

**这是本模块最大的性能陷阱。** `INTER_AREA` 的开销由**源像素数**决定：它要把整个
ROI 读一遍做面积平均。只取下部区域省下的时间会被它全部吃掉。实测同一 ROI：

| 插值方式 | 耗时 |
| --- | ---: |
| `INTER_AREA` | 0.319 ms |
| `INTER_LINEAR` | 0.021 ms |
| `INTER_NEAREST` | 0.010 ms |

采用 `INTER_LINEAR`。红线有 50 像素以上宽，通道差分阈值又极具选择性，缩放时的
抗锯齿没有实际价值。

## 列投影与双阈值 run 提取

**带是水平的**：整幅宽、12 行高的横条。列投影指沿竖直方向把 12 行求和压成 1 行，
得到 160 个数，第 `i` 个数是第 `i` 列中红色像素的个数。结果按列索引，即按 x 坐标
排开。实现沿第 0 维（行）求和。

**不能使用 `cv2.reduce`**：CanMV 固件上的精简版 OpenCV 没有这个函数，上板会直接抛
`AttributeError`。改用 `np.sum(band_mask, axis=0)`，实测在桌面上也比 `cv2.reduce`
更快（单帧 0.185 ms → 0.112 ms）。不同固件对 `axis` 参数的支持不一致，构造函数会
探测一次并缓存结果；不支持时退回逐行相加，两条路径在 29 张实拍图上结果完全一致。

得到的一行**必须先转成 Python 列表再遍历**。逐个索引数组元素每次都要装箱，实测
比先 `tolist()` 慢约 3 倍，在解释执行的板端差距只会更大。

扫描使用两个阈值，一次遍历同时得到两组信息：

| 阈值 | 默认值 | 用途 |
| --- | ---: | --- |
| `LINE_MAIN_MIN_COLUMN_COUNT` | 8 | 提取主线 run |
| `LINE_EDGE_MIN_COLUMN_COUNT` | 2 | 测量红色向左右延伸到哪里 |

**双阈值是本设计的关键**：主线基本竖直，会填满整条带的 12 行，列计数接近 12；
路口横线只占带内很少几行，列计数只有 3~5。用高阈值提取 run 时横线不会并进主线，
**主线中心在路口不会被拽偏**。实测这一项同时修好了十字图里 `b2` 偏差被横线带偏
的问题（36 → 14）。

边缘阈值必须取得很低。横线压在带边界上时在这条带里可能只占一两行；阈值过高就会
出现「计入了红色总量却测不出横向范围」的自相矛盾结果，表现为判出路口却报不出
左右分支。

run 提取规则：

1. 列计数达到对应阈值的列视为被占用。
2. 空隙不超过 `LINE_RUN_MAX_GAP` 时合并为同一个 run。
3. 宽度小于 `LINE_RUN_MIN_WIDTH` 的 run 丢弃。
4. 宽度超过 `画面宽 × LINE_RUN_MAX_WIDTH_RATIO` 的 run 不作为主线中心。

**不使用整行质心**。十字路口的横向分支会把质心拽偏，run 不会。

多个 run 时的选择：带 0 选最靠近画面中心的 run，其余各带选中心最接近上一条带所选
run 的那个。近带锚定主线，横向分支抢不走远带。

## 偏差与置信度

- `offset_i = 画面中心X − 带 i 的 center_x`，单位是原图像素，符号与现有全部模块
  一致：线偏左则为正，车该左转。
- `confidence`：`有效带比例 × 0.6 + 带间连续性 × 0.4`。连续性为
  `1 − min(1, 平均相邻带中心跳变 / (画面宽 × LINE_CONTINUITY_REF_RATIO))`。
  该值只用于本模块内部判断，不与其他检测器横向比较。

有效带数少于 `LINE_MIN_VALID_BANDS` 时 `detect()` 返回 `None`。

## 路口判定

复用已有的列投影统计，不调用 `RoadSymbolDetector`，稳态零额外开销。

```text
预期红量 = 各带主线 run 宽度的中位数 × 带高
逐带超量比 = 该带红色像素数 / 预期红量
```

判为路口的条件是**逐带超量比的最大值** ≥ `LINE_JUNCTION_MASS_RATIO`，且异常带数
≥ `LINE_JUNCTION_MIN_BANDS`。

三条关键性质：

1. **比值必须逐带取最大，不能把 5 条带加总。** 横线通常只落在一到两条带里，求和
   会把它稀释 5 倍。初版就是错在这里，召回为 0。
2. **倾斜不影响。** 横线不管怎么斜，总要横穿整个视野；斜只是把红色像素重新分配到
   不同的带里。
3. **弯道不误判。** 弯道时主线自身变宽，中位数同步抬高，分子分母一起变大，比值仍
   在 1 附近。

异常带的统计**不要求该带有主线**：T 型路口横线所在的那条带主线已经到头，带内只剩
横穿的红色，这正是最典型的路口带。

`junction_flags:u8` 按位定义（`line.py` 与 `uart_io.py` 各存一份，必须一致）：

| 位 | 含义 |
| --- | --- |
| bit0 | 接近路口 |
| bit1 | 左侧存在分支 |
| bit2 | 右侧存在分支 |
| bit3 | 丢线 |
| bit4~7 | 保留，固定为 0 |

左右分支必须在**所有超量带**上累计，不能只看比值最大的那一条。倾斜的横线会被带
边界切成两段落进相邻两条带，其中一段只向左、另一段只向右延伸，只看一条带会漏掉
另外半边。

该标志只负责「接近路口了，减速并准备切数字识别」，不承担精确的 T 型与十字分类。
精确分类仍应交给 `road.py`。

## 绘制

`detect()` 不修改输入画面。`draw()` 分两层，数据层可以整体关闭：

**几何层**（始终绘制，实测 0.028 ms）：

- 5 条带的边界线和画面中心竖线，用于目测偏差。
- 每条带所选 run 的中心点，有效为绿色，无效为红色。
- 相邻有效带中心之间的连线，即拟合出的路线。

**数据层**（由 `draw_data` 和 `draw_band_labels` 控制，实测 0.060 ms）：

- 左上角三行：五条带偏差、超量比及其所在带、置信度、有效带数、标志位及其展开
  文本。无效带显示 `--`，不显示 0，避免与「偏差为 0」混淆。
- 每条带中心点旁标出该带自己的偏差，贴近画面右边界时自动改到点的左侧。
- 丢线帧由调用方调用 `draw_lost()` 显示 `LINE LOST valid=0`。`draw()` 在
  `result` 为 `None` 时仍然直接返回，不承担状态显示。

数据层每帧多做八次 `putText`，占检测加绘制总耗时的约 30%（0.060 / 0.198 ms），
**实际上车应关闭**。终端周期打印复用同一个 `format_result()`，保证画面和日志
格式一致，便于对照。

## 串口协议

沿用现有帧格式，走同一个 `send_frame()`，不改握手流程，不改 TARGET：

```text
AA 55 | VER | TYPE | SEQ | LEN | PAYLOAD | CRC8
```

新增 `UART_MESSAGE_LINE = 0x11`，PAYLOAD 定长 12 字节，整帧 19 字节：

```text
valid:u8 | b0:int16_LE | b1:int16_LE | b2:int16_LE | b3:int16_LE | b4:int16_LE
        | junction_flags:u8
```

- `b0` 最近（车头正前方，供 TI 板 PID 使用），`b4` 最远（前瞻）。
- 单条带无效时该字段发 0，`junction_flags` 不置 bit3。
- `valid=0`（整体丢线）时全部偏差强制为 0，`junction_flags` 置 bit3。TI 板不得把
  `0` 解释为「红线位于画面中心」，必须走丢线保护逻辑。

`send_line()` 的周期控制、`force` 参数和返回值语义与 `send_target()` 完全一致。

保留 `UART_MESSAGE_DIGIT = 0x12` 给后续病房号上报，本次不实现。

带宽占用：19 字节 × 30 FPS = 570 字节/秒，约为 115200 波特的 5%。

## 主程序状态机

```text
TRACKING：每帧 process() → send_line()
          junction_flags bit0 连续 LINE_JUNCTION_CONFIRM_FRAMES 帧置位 → 切 DIGIT
DIGIT：   本次留 TODO 和明确接入点，不实现
```

连续帧确认由 `JunctionConfirmState` 负责。它只服务于主程序的状态切换，不属于
检测器，也不会把历史结果当成当前帧的检测结果发送。

`enable_uart=False` 时跳过 `TrackingUART` 的创建和握手，只做视觉和显示，便于单独
调参。默认 `True`，启动时阻塞等待 `wait_for_handshake()`。

主循环在循环外创建 `time.clock()`，每帧 `tick()`，在 `show_image()` 前把 FPS 画到
画面上，与 `road.py` 的演示保持一致。

## 实拍回归结果

样本为 `pic/line` 的 29 张 `640×480` 实拍图（`pic/` 在 `.gitignore` 中，不进版本
库）。人工标签：T 型 14 张（0–13）、十字 11 张（14–24）、终点虚线 4 张（25–28）。

| 指标 | 结果 |
| --- | --- |
| 路口召回 | **25/25** |
| 终点误报路口 | **0/4** |
| 直道误报路口 | **0/29** |
| 返回 `None` | 0/29 |
| 左右分支位正确 | 28/29 |

三类场景的超量比完全分开，阈值 `1.50` 落在空隙中间：

```text
直道   1.00 ~ 1.22
终点   1.05 ~ 1.35
        ↑ 阈值 1.50 ↓
路口   1.77 ~ 6.06
```

**直道样本是从实拍图构造的**：所有实拍图的横线都在 `0.35`~`0.63`，把 ROI 限制到
`0.70`~`1.00` 后视野里只剩一条竖直主线，等价于直道场景。这一步是必要的——原始
29 张图里**没有一张纯直道**，而「直道不误报路口」恰恰是最重要的负样本。

唯一不完美的是第 19 张：它的横线在 `0.27`~`0.33`，几乎整条都在 ROI 之外，只报出
路口和右分支，没报左分支。这是正确行为，车再靠近横线就会进入 ROI。

耗时对比（桌面 CPython，绝对值不代表 K230，相对关系用于确认实现没写反）：

| 检测器 | 单帧耗时 |
| --- | ---: |
| `LineTrackDetector` | 0.169 ms |
| `RoadSymbolDetector` | 0.377 ms |

串口帧编码用假 `machine` 模块验证：帧头、版本、类型、LEN、CRC8 和 int16 小端
编码全部正确，覆盖有效帧、丢线帧、带数不足补零和真实检测结果四种情况。

## 待补验证

1. **板端帧率**。桌面耗时不代表 K230。上板运行 `line.py` 演示，记录画面上的 FPS。
   若未明显高于 `road.py` 演示，优先检查是否误用了 `INTER_AREA`。
2. **更多光照条件**。现有 29 张实拍图光照接近，`LINE_RED_MIN_DIFF` 在强光和阴影
   下需要复测。
3. **真实直道图**。当前直道负样本是构造的，仍应补拍无路口的直道图确认。
4. **弯道图**。现有实拍图中没有明显弯道，「弯道不误判」目前只有推理支持，缺实测。

## 已知取舍

- 通道差分对偏橙或偏粉的地面反光敏感，阈值需现场标定。
- 不做时间平滑，单帧丢线立即上报 `valid=0`，抗抖动责任在 TI 板。
- 路口判定不区分 T 型和十字。要精确分类需在触发后调用 `road.py`。

## 设计过程中被推翻的假设

初版设计里下面几条经实拍标定后证明是错的，记录下来避免重复：

1. **「红线约 22 像素宽」** —— 实测底部 52~67 像素。因此主线自身的红色量远比预期
   大，初版估算的路口超量比 4.4 倍实际只有 1.1~1.8 倍。
2. **「把 5 条带的红色加总作为路口判据」** —— 稀释了信号，召回 0/25。必须逐带
   取最大。
3. **「带之间留 0.02 的间隙」** —— 间隙会让水平横线漏检。改为整体缩放一次、带间
   无空隙。
4. **「处理像素少 8 倍就会快 8 倍」** —— 忽略了 `INTER_AREA` 的开销由源像素数
   决定。修正前反而比 `road.py` 慢。
5. **「run 宽度暴增是路口的主判据」** —— 倾斜 45° 时宽度只剩 1.7 倍，会漏判。
   改为红色超量比，倾斜无关。
6. **「按行统计能找到横线」** —— 第 16、19 两张图的横线是斜的，没有任何一行的红色
   超过画面宽度的 50%，按行统计直接漏掉。
